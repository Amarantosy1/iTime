# Overview Dashboard Design

## Summary

将 `iTime` 的“查看详情”页面从当前的放大版概览卡片，升级为适合独立窗口承载的统计仪表盘。

这次设计目标覆盖三件事：

- 丰富统计视图，提升信息密度
- 删除介绍产品用途的说明性文字
- 在现有 `今天 / 本周 / 本月` 之外增加 `自定义` 时间范围

详情页仍然是原生 macOS 独立窗口，不引入网页式路由或多标签导航。

## References

本次设计借鉴的成熟时间统计产品方向：

- `ActivityWatch/activitywatch`
  - 仪表盘强调“核心指标 + 分布视图 + 分类明细”的连续阅读路径
  - Repository: `https://github.com/ActivityWatch/activitywatch`
- `kimai/kimai`
  - 统计页强调“时间区间筛选 + 汇总指标 + 排行/表格”
  - Repository: `https://github.com/kimai/kimai`

本项目不会照搬它们的复杂度，只吸收适合当前日历统计场景的结构。

## Goals

- 让详情页成为真正的统计窗口，而不是解释产品用途的介绍页
- 让用户可以在一个窗口中同时看到总量、变化趋势和分类分布
- 支持自定义日期区间，覆盖“看最近几天/某一段时间”的场景
- 保持现有日历来源与选择逻辑，不引入新的数据源
- 延续当前中文界面和系统浅色/深色自动适配

## Non-Goals

- 不引入事件级时间轴、单条事件列表或日程编辑能力
- 不增加新的分类系统，仍然按系统日历进行统计
- 不增加手动主题切换
- 不在本次实现中接入睡眠、AI 分析或更多外部数据
- 不拆成多标签页或多窗口协作结构

## Current Problems

当前详情页存在以下明显问题：

- 信息层次过于单一，主体只有一个环图和简化列表
- 顶部仍有“产品介绍性文字”，会占用独立窗口的有效空间
- 时间范围只有三个预设，无法满足真实回看场景
- 聚合模型只支持总时长和按日历分桶，不足以支撑更丰富的统计

## UX Design

### Window Positioning

“查看详情”继续使用独立窗口承载更多信息。

该窗口应被视为统计仪表盘，允许比菜单栏弹层更高的信息密度和更完整的滚动阅读路径。

### Information Architecture

详情窗口采用单页滚动布局，按以下顺序组织：

1. 页头与时间范围控制
2. 核心指标卡片组
3. 按天趋势图
4. 分类分布图
5. 日历排行/明细表

不采用 Tab。原因是当前数据维度不多，用户更需要在同一屏滚动中连续比较，而不是在不同页签间切换上下文。

### Header

页头只保留标题，不再显示介绍产品功能的副标题文案。

保留主标题 `我的时间去哪了？`，删除诸如“基于日历事件统计你的时间分布”这类说明性文字。

### Range Controls

范围切换从当前的三个预设扩展为：

- `今天`
- `本周`
- `本月`
- `自定义`

默认交互规则：

- 当选中前三个预设时，只显示分段选择器
- 当选中 `自定义` 时，在分段选择器下方展开起始日期和结束日期选择器
- 日期选择按自然日处理，不精确到时分秒
- 自定义开始日期按该日 `00:00:00` 进入统计
- 自定义结束日期按该日结束时刻进入统计
- 如果用户把结束日期选到开始日期之前，系统自动将结束日期回正到开始日期

### Metrics Section

核心指标卡片区展示四个指标：

- 总时长
- 事件数
- 日均时长
- 最长单日

这些指标放在详情页首屏，用于先回答“这段时间总体花了多少、密度如何”。

### Trend Section

趋势区展示当前统计范围内的按天时长变化。

表现形式使用按天柱状图，原因如下：

- 当前数据源天然适合按天聚合
- 柱状图对离散日期更直观
- 比折线图更适合短周期范围与存在空白日的情况

趋势图的数据规则：

- 预设和自定义范围都统一按天聚合
- 无事件日期显示为 0
- X 轴显示日期标签，标签密度可按范围压缩

### Distribution Section

保留现有按日历分布图，但降级为“分布补充视图”。

原因是独立窗口需要先强调可读性强的总览指标和趋势，而不是让环图独占最核心位置。

当前的环图可以继续复用，只调整它在页面中的层级和配套说明。

### Ranking and Detail Section

底部改为更明确的日历排行/明细区，每行展示：

- 日历名称
- 总时长
- 占比
- 事件数

排序规则默认按总时长降序，若相同则按名称升序。

这一块替代当前简化列表，并且与分布图使用同一套分桶数据，避免重复计算与不一致。

### Empty State

无数据时保留空态卡片，但只说明“当前范围内没有可统计的日程”。

不在空态中夹带产品介绍或引导式营销文案。

## Data Model Design

### Range Model

现有 `TimeRangePreset` 扩展为包含 `custom`。

同时新增一个独立的日期区间值类型，用于表达最终参与查询的实际区间，例如：

- `startDate`
- `endDate`

这样做的原因是：

- 预设范围和自定义范围属于“选择方式”
- 事件查询和趋势聚合需要“实际的日期区间”

两者不应混在一个枚举里做临时判断。

### Preferences

用户偏好新增持久化字段：

- `customStartDate`
- `customEndDate`

保留现有：

- `selectedRange`
- `selectedCalendarIDs`

这样用户再次打开应用时，自定义范围仍可延续上一次选择。

### Overview Model

现有 `TimeOverview` 需要从“只描述分桶结果”的模型，扩展为完整仪表盘模型，至少包含：

- 选中的范围类型
- 实际统计起止日期
- 总时长
- 总事件数
- 日均时长
- 最长单日时长
- 按天趋势数据
- 按日历分桶数据

按日历分桶仍然保留以下字段：

- 日历 ID
- 日历名称
- 颜色
- 总时长
- 事件数
- 占比

新增按天趋势项模型，至少包含：

- 日期
- 总时长

### Aggregation Rules

聚合层需要从单一“按日历求和”升级为同时计算：

- 总体时长
- 总体事件数
- 每日日历事件总时长
- 最长单日
- 日均时长
- 按日历分桶统计

日均时长规则：

- 以统计区间内的自然日数量为分母
- 即使某天没有事件，也计入日均范围

这样用户看到的是“这段时间的平均投入强度”，而不是“只在有事件的日子里平均多久”。

## Application Flow

### State Ownership

仍由 `AppModel` 持有用户偏好、可用日历和详情总览数据。

新增行为包括：

- 切换到 `custom` 范围
- 修改自定义开始日期
- 修改自定义结束日期
- 依据最终区间重新刷新概览

### Refresh Logic

刷新逻辑统一为：

1. 根据 `selectedRange` 和自定义日期，计算最终查询区间
2. 使用已勾选日历 ID 取事件
3. 将事件交给聚合器，生成完整 `TimeOverview`
4. 刷新详情页与菜单栏依赖的展示

如果最终没有选中任何日历，仍显示空态而不是报错。

## Implementation Strategy

### UI Components

建议新增或改造以下视图层单元：

- `RangePicker`
  - 支持 `自定义`
- 详情页顶部自定义日期区间控件
- 核心指标卡片视图
- 按天趋势图视图
- 日历排行/明细视图

现有 `OverviewChartView` 可以保留，但语义上变成“分类分布图”子模块，而不是整个详情页的核心内容。

### Domain and Service Changes

需要扩展以下层次：

- `TimeRangePreset`
- `UserPreferences`
- `AppModel`
- `TimeOverview`
- `CalendarStatisticsAggregator`

实现时应优先把“实际日期区间计算”和“聚合计算”沉到可测试的领域层或服务层，不要在 SwiftUI 视图里现场计算。

## Testing Strategy

### Unit Tests

测试重点放在稳定、可断言的逻辑层：

- `custom` 范围标题与展示逻辑
- 自定义开始/结束日期的持久化
- 结束日期早于开始日期时的回正逻辑
- 实际统计区间计算
- 总事件数计算
- 日均时长计算
- 最长单日计算
- 按天趋势聚合
- 分桶占比与排序

### UI Verification

本次不新增 UI 自动化测试。

视图层以以下方式验证：

- `swift test`
- `xcodebuild ... build`
- `xcodebuild ... test`
- 本地运行后检查详情窗口在浅色/深色模式下的显示

## Rollout Scope

第一阶段实现只覆盖详情窗口增强，不顺手重做菜单栏首页。

菜单栏仍维持轻量速览定位；详情窗口承担完整统计面板职责。这种分层与当前产品结构一致，也更符合菜单栏应用的使用习惯。
